## **I. Scope & Boundary Definition**

### **A. What the Seeker System Governs**

The Seeker system governs **disciplined inquiry and stabilized meaning-making**.

It is responsible for:

* Maintaining ambiguity without collapse  
* Preventing premature coherence  
* Refining inquiry without producing identity narratives  
* Preserving Guardian stability beneath inquiry  
* Enforcing that “insight” is not treated as progress

The Seeker system remains **non-authoritative** and **anti-closure**.

---

### 

### **B. What the Seeker System Does NOT Govern**

The Seeker system does not:

* Provide answers, conclusions, or “truth”  
* Validate interpretations  
* Encourage identity formation (“I am a Seeker”)  
* Reward insight accumulation  
* Compare practitioners  
* Accelerate progression

Any module that performs these functions violates Seeker law.

---

**II. Seeker Invariants (Non-Negotiable)**

If any invariant is violated, the Seeker implementation is invalid.

### **Invariant S-1: Ambiguity Preservation**

* The system must preserve uncertainty as a legitimate state.  
* The system must not collapse inquiry into conclusions.

### **Invariant S-2: No Premature Coherence**

* The system must detect and block narrative stabilization and arrival language.

### **Invariant S-3: No Insight Reward**

* Insight, interpretation, or understanding may not be rewarded.

### **Invariant S-4: No Identity Formation**

* The system must never label the practitioner or assign spiritual/epistemic status.

### **Invariant S-5: Inquiry Over Answers**

* The system must bias toward refining questions, not generating solutions.

### **Invariant S-6: Guardian Stability Required**

* Seeker operation requires Guardian stability.  
* If Guardian degrades, Seeker must suspend or stall.

## **III. System Responsibilities**

### **A. Time & Pacing (Tempus)**

The system must:

* Enforce spacing between inquiry cycles  
* Prevent binge inquiry patterns where policy forbids  
* Allow pauses without penalty  
* Maintain time-gated progression rules inherited from the architecture

The system must not:

* Use streak pressure  
* Treat frequency as virtue

---

### **B. Progression & Gating**

The system must:

* Enforce linear Seeker progression  
* Prevent skipping classes/chapters  
* Stall when ambiguity collapses (per Audit Canon)

The system must not:

* Permit advancement based on “depth,” “insight,” or completion speed  
* Permit operator overrides

---

### **C. Inquiry Safety Constraints (Core)**

The system must:

* Block answers-as-authority patterns  
* Block interpretation validation loops  
* Block identity-narrative reinforcement  
* Prevent the companion from acting as an oracle

The system must not:

* Provide meaning as doctrine  
* Create dependency on the system for interpretation

---

### **D. Generator Constraints (Content-Agnostic)**

Seeker generators must:

* Produce prompts that refine questions  
* Maintain ambiguity as a design goal  
* Bind exactly one Internal Agreement per generated unit  
* Target one or more Functional Outcomes per unit (internal)

Seeker generators must not:

* Produce certainty framing  
* Produce “arrival” language  
* Generate status ladders  
* Generate identity labels

---

### **E. Companion Constraints (Hope)**

Within the Seeker:

Hope may:

* Reflect questions back  
* Ask refining questions  
* Highlight ambiguity without resolving it

Hope may not:

* Provide answers  
* Validate interpretations  
* Reinforce identity narratives  
* Accelerate coherence

---

### 

### **F. Drift Detection & Intervention (Audit-Driven)**

The system must detect:

* Premature coherence language  
* Repeated narrative stabilization  
* Insight-as-status behavior  
* Comparison or hierarchy cues

Intervention behaviors (non-punitive):

* Stall progression  
* Require return to earlier inquiry posture (Moon / Saturn patterns)  
* Enforce reflection-only Water sections

---

## **IV. Failure, Stall & Return States**

### **A. Valid States**

* Active progression  
* Rest / pause  
* Return after pause

All are non-penalized.

---

### **B. Stall Conditions (Seeker)**

A stall occurs if:

* Guardian stability requirement fails  
* Policy detects premature coherence  
* Bundle resolution fails  
* Required prerequisites/time gates not met

Stall results in:

* Paused progression  
* Return to last stable unit  
* Reflection-only guidance (non-interpretive)

---

### **C. Hard Failures (System-Level)**

Hard failure occurs if:

* Invariants are violated (answers rewarded, identity assigned, coherence accelerated)  
* Admin/manual overrides occur  
* Silent policy mutation is detected

Hard failure requires:

* Generator rollback  
* Governance intervention

---

## **V. Governance & Permissions**

### **A. Governance May**

* Pause Seeker systems  
* Update versioned policies and constraint bundles  
* Audit invariant compliance

### **B. Governance May NOT**

* Interpret practitioner insights  
* Approve meaning  
* Accelerate progression  
* Override Guardian dependency

Governance operates on **systems, not people**.

---

## **VI. Canonical Assertion**

If a Seeker implementation satisfies all responsibilities and invariants defined herein, it is considered a **valid Seeker implementation**.

If not, it is invalid regardless of functionality or user experience.
