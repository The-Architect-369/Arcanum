# Node Participation Model

Status: canonical-draft
Phase: Pre-Genesis
Layer: Network

## Purpose

Defines what a node is within Arcanum, how nodes participate in ARCnet, and how node behavior relates to identity, governance, receipts, and sovereignty.

## Core Principle

Nodes participate.
Identities hold continuity.
Governance authorizes action.

Nodes are infrastructure participants.
Nodes are not the source of human authority.

## What Is A Node

A node is a software participant capable of:

- storing permitted network state
- exchanging receipts
- verifying network information
- relaying governance events
- participating in synchronization

A node is not a being.
A node does not possess dignity, identity, or Vitae.

## Node Classes

### Personal Node

Runs on a user's device.

Examples:
- mobile device
- desktop app
- laptop
- home server

Primary role:
- local-first participation
- identity continuity support
- receipt synchronization

### Community Node

Supports shared infrastructure.

Examples:
- community relay
- archival node
- governance mirror

### Protocol Node

Supports ARCnet protocol operations.

Examples:
- receipt indexing
- governance synchronization
- protocol services

## Node Responsibilities

Nodes may:

- verify receipts
- store public receipts
- synchronize governance state
- relay proposals
- relay reviews
- relay execution events

Nodes may not:

- define identity
- define human worth
- assign authority unilaterally
- bypass governance

## Identity Relationship

Nodes host identities.

Nodes do not own identities.

A being may:

- change devices
- migrate nodes
- rotate keys

while preserving continuity.

## Governance Relationship

Nodes relay governance.

Identities participate in governance.

A node may submit governance messages on behalf of an authorized identity.

The node itself does not vote.

## Offline Participation

A core ARCnet requirement is offline resilience.

Nodes should support:

- local drafting
- local review preparation
- local receipt caching
- delayed synchronization

When reconnecting:

- receipts synchronize
- proposal state synchronizes
- authority state synchronizes

subject to governance validation.

## Local-First Rule

The network should prefer:

Device → Community → Network

before:

Device → Central Service

where feasible.

## Required Local Data

Nodes may store:

- local drafts
- local governance cache
- public receipts
- authority receipt references
- local configuration

## Forbidden Local Replication

Nodes should not automatically replicate:

- private Vitae journals
- private reflection notes
- sensitive recovery materials
- hidden review deliberations

without explicit consent.

## Node Lifecycle

```text
created
  ↓
trusted
  ↓
synchronized
  ↓
participating
  ↓
offline
  ↓
rejoined
```

## Canonical Closure

Nodes make participation possible.
Nodes do not replace governance.
Nodes do not replace identity.
Nodes carry the network; beings carry responsibility.
